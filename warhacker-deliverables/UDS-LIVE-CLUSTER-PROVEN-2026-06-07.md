## đź”§ The fix recipe (the underlying REAL repo bugs â€” now fixed in `tasks.yaml`)

The repo's one-button `uds run start` task did **not** work as written. The path that works (now automated):

1. **`start` task is internally inconsistent** â€” it builds the root monolithic `zarf.yaml` then tries to deploy a bundle named `szl-warhacker-demo` that the build doesn't produce. **Use the modular package instead:** `packages/szl-receipts/zarf.yaml` (it deliberately excludes pepr/ui) + the root `uds-bundle.yaml`.
2. **Pepr build needs `--legacy-peer-deps` + 8 undeclared deps**, and the Pepr **module/component name** in the manifest didn't match (`szl-receipt-policy`). Fix both, then `npx pepr build` produces `dist/pepr`.
3. **Removed the `szl-ui-dashboard` component** â€” its `ui/` chart doesn't exist in the repo (build fails referencing it).
4. **Defense Unicorns registry now needs auth.** `registry.defenseunicorns.com/.../uds/core` returns **401**. Repoint Core to the public mirror: **`ghcr.io/defenseunicorns/packages/uds/core:1.5.0-upstream`** (anon-pullable). *(The lighter `k3d-core-slim-dev` bundle is **403 / not anon-pullable** â€” you can't use it without DU credentials.)*
5. **Bundle override key mismatch** â€” the override targeted component `szl-receipts`; the real component is **`szl-receipts-server`**. Fix the key or the override is silently ignored.
6. **Signature validation** â€” init, core, and the receipts package are keyless-signed by **different identities**, so add **`--skip-signature-validation`** to `uds deploy` (a demo expedient, NOT a security posture â€” say so if asked).

## đź§± Two HARD ceilings found â€” read before choosing the demo machine

These are the surprises we did NOT want Stephen to hit on stage. Both are about the **full upstream UDS Core**, not about Stephen's code.

### Ceiling 1 â€” full UDS Core does not fit 2 vCPU
After Core's Istio + Pepr came up, the node was at **2000m / 2 cores = 100% CPU requested.** The **tenant ingress gateway** then sat `Pending` with `Insufficient cpu`, and **Keycloak + Prometheus** (still to come) would not have fit either. The receipts server only fit after trimming CPU requests on a few Core components.
**Implication:** the **tower (multi-core) is the right machine** for a full Core boot. A 2-vCPU box (like this one, or a small cloud VM) **cannot** run stock upstream Core in full.

### Ceiling 2 â€” stock Core ships TWO LoadBalancer gateways; single-node k3d can't host both
Core defines an **admin** gateway and a **tenant** gateway, both `LoadBalancer`. On single-node k3d, the built-in klipper load-balancer gives the **first** gateway the node IP and host 80/443; the **second** gateway's load-balancer then fails with **"no free ports"** and never gets an IP.
**Implication â€” this would bite the tower too, not just this box.** A single-node k3d cluster (which is what a laptop/tower demo uses) cannot stand up **both** stock gateways. Options for Stephen, in order of safety:
- **Don't depend on the tenant gateway for the demo.** The receipts server is reachable via `kubectl port-forward` (that's exactly how it was proven tonight) â€” no gateway needed.
- Use the **slim-dev Core profile** (single gateway, smaller footprint) â€” but that bundle needs **DU credentials** (403 anon).
- Configure the **k3d load-balancer with explicit port mappings** / a single gateway override at cluster-create time.
